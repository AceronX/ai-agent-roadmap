# Embeddings and Vector Search

## Goal

Understand how text can be converted into vectors, how similar vectors are searched, and how this becomes the retrieval layer behind RAG, semantic search, long-term memory, and knowledge tools for AI agents.

By the end of this topic, you should be able to answer:

- What is an embedding?
- Why is vector search different from keyword search?
- What information must be stored with each vector?
- How does an agent use retrieved chunks safely?
- What common mistakes make retrieval quality poor?

## Why It Matters

LLMs only know what is inside their model weights and current context window. They do not automatically know your private documents, tickets, database records, meeting notes, codebase, or latest business policies.

For an AI agent, embeddings and vector search are the bridge between:

- the user's current request,
- external knowledge,
- previous memories,
- tool outputs,
- and the final answer generated by the LLM.

Without retrieval, an agent often has to guess. With retrieval, the agent can look up relevant evidence before answering or acting.

## What Is an Embedding?

An embedding is a list of numbers that represents meaning.

```text
"How do I reset my password?"
  -> [0.021, -0.338, 0.104, 0.872, ...]
```

Each number is one dimension of the vector. Real embedding models often produce hundreds or thousands of dimensions, such as 384, 768, 1536, or 3072 dimensions depending on the model.

You do not manually decide what each dimension means. The model learns useful dimensions during training. The practical result is:

- similar text produces nearby vectors,
- unrelated text produces distant vectors,
- search can compare meaning instead of only words.

## What Can Be Embedded?

Embeddings are not only for individual words. In agent systems, you usually embed larger units:

- a user query,
- a paragraph from a document,
- a support-ticket summary,
- a code function or file section,
- a tool result,
- a conversation memory,
- an image or audio item if using multimodal embeddings.

For text RAG, the most common unit is a chunk: a small piece of a larger document.

## Why Not Embed Whole Documents?

Large documents often contain many unrelated ideas. If you embed the entire document, the vector becomes a blurry average of all topics inside it.

Bad retrieval unit:

```text
100-page employee handbook
```

Better retrieval units:

```text
Paragraph about password policy
Paragraph about vacation approval
Paragraph about expense reimbursement
Paragraph about laptop return process
```

Good chunking helps the vector represent one specific idea instead of many mixed ideas.

## Chunking

Chunking is the process of splitting documents before embedding them.

A practical starting point for text documents:

- chunk size: 300-800 tokens,
- overlap: 50-150 tokens,
- keep headings and section titles with the chunk,
- avoid cutting tables, code blocks, or procedures in the middle,
- store source metadata for citation and permissions.

Example:

```text
Original document:
  Company IT Handbook

Chunks:
  1. Account setup
  2. Password reset
  3. MFA recovery
  4. Device replacement
```

Chunking is often more important than the choice of vector database. If the relevant answer is split badly or mixed with unrelated text, search quality will suffer even with a strong embedding model.

## What Is Vector Search?

Vector search finds stored vectors that are closest to a query vector.

```text
Query:
  "How can I recover my account password?"

Query vector:
  [0.13, 0.44, 0.22, ...]

Nearest stored vectors:
  1. Password Reset Guide
  2. Account Recovery Policy
  3. Login Troubleshooting
```

The vector database does not "understand" the answer like an LLM. It compares numeric vectors and returns the closest matches.

## Similarity Metrics

A similarity metric defines what "close" means.

Common metrics:

- Cosine similarity: compares vector direction; common for text embeddings.
- Dot product: compares direction and magnitude; often used when vectors are normalized.
- Euclidean distance: compares straight-line distance between points.

In most beginner RAG systems, use the metric recommended by the embedding model or vector database integration. Do not mix embeddings generated for one metric with an index configured for a different metric unless you understand the effect.

## Exact Search vs Approximate Search

With a small number of vectors, you can compare the query against every stored vector.

```text
Exact search:
  compare query with every vector
  -> accurate but slower at large scale
```

At large scale, vector databases usually use approximate nearest neighbor search.

```text
Approximate search:
  use an index structure
  -> much faster
  -> may miss some perfect matches
```

This creates a tradeoff:

- higher recall usually costs more latency and memory,
- lower latency may miss relevant chunks,
- index settings matter more as data grows.

For learning projects, exact search or a simple local vector index is fine. For production, use a vector database or search engine that supports indexing, filtering, persistence, monitoring, and access control.

## What To Store In A Vector Database

A vector alone is not enough. Store the vector with payload metadata.

Example record:

```json
{
  "id": "it-handbook-password-reset-003",
  "text": "To reset your password, open the identity portal...",
  "embedding": [0.021, -0.338, 0.104],
  "metadata": {
    "source": "IT Handbook",
    "url": "https://intranet.example.com/it/password-reset",
    "section": "Password Reset",
    "created_at": "2026-02-10",
    "tenant_id": "company-a",
    "visibility": "employees"
  }
}
```

Important metadata:

- source title,
- source URL or file path,
- page, heading, or line number,
- document version,
- tenant or user access scope,
- timestamps,
- tags or categories,
- language,
- original document ID.

Metadata lets the system filter results, show citations, enforce permissions, and debug bad answers.

## Retrieval Pipeline

A typical ingestion pipeline:

```text
Documents
  -> clean and normalize text
  -> split into chunks
  -> embed each chunk
  -> store vector + text + metadata
  -> build or update vector index
```

A typical query pipeline:

```text
User question
  -> embed question
  -> apply metadata filters
  -> retrieve top-k similar chunks
  -> optionally rerank chunks
  -> place best chunks in LLM context
  -> generate answer with citations
```

For an agent, retrieval can be a tool:

```text
Agent receives task
  -> decides it needs company knowledge
  -> calls search_knowledge_base(query)
  -> reads retrieved chunks
  -> answers or calls another tool
```

## Top-K And Score Thresholds

`top_k` controls how many results are returned.

```text
top_k = 3   -> faster and cheaper, but may miss context
top_k = 20  -> more evidence, but more noise and tokens
```

A score threshold filters out weak matches.

```text
Only return chunks with similarity >= 0.75
```

Be careful: similarity scores are not universal. A score of `0.75` may be strong for one model and weak for another. Calibrate thresholds with your own queries and documents.

## Metadata Filtering

Vector similarity alone is not enough for real applications.

Example:

```text
Query: "What is the refund policy?"
```

Without filters, the system might retrieve:

- refund policy for the wrong country,
- old policy from last year,
- admin-only policy,
- policy from another customer tenant.

Add filters:

```text
country = "Germany"
document_status = "published"
tenant_id = "acme"
visibility in ["public", "customer"]
```

For agents, permission filters are not optional. Never retrieve private chunks and then rely on the LLM to ignore them. Filter before retrieval or inside the vector database query.

## Reranking

Initial vector search is usually optimized for speed. A reranker can take the top 20-100 candidates and reorder them with a more precise model.

```text
Vector search:
  fast, broad recall

Reranker:
  slower, better precision
```

Use reranking when:

- many chunks look similar,
- answers require exact evidence,
- top result quality is inconsistent,
- the domain has long or technical documents.

## How This Helps AI Agents

Embeddings and vector search are used in several agent capabilities.

Knowledge retrieval:

```text
Find relevant documentation before answering.
```

Long-term memory:

```text
Store facts from previous interactions and retrieve them later.
```

Tool selection:

```text
Embed tool descriptions and retrieve the tools most relevant to the task.
```

Example selection:

```text
Retrieve similar solved tickets, code snippets, or workflows.
```

Planning support:

```text
Retrieve prior plans, constraints, and project notes before choosing actions.
```

The agent should not blindly trust retrieved text. It should use retrieval as evidence and still reason about whether the evidence answers the actual user request.

## Minimal Example

This simplified example shows the shape of vector search without a vector database.

```python
from numpy import dot
from numpy.linalg import norm


def cosine_similarity(a, b):
    return dot(a, b) / (norm(a) * norm(b))


documents = [
    {
        "id": "doc-1",
        "text": "Reset your password from the identity portal.",
        "embedding": [0.90, 0.10, 0.05],
    },
    {
        "id": "doc-2",
        "text": "Submit expenses within thirty days.",
        "embedding": [0.05, 0.85, 0.10],
    },
    {
        "id": "doc-3",
        "text": "Use multi-factor authentication when logging in.",
        "embedding": [0.75, 0.20, 0.15],
    },
]

query_embedding = [0.88, 0.12, 0.04]

ranked = sorted(
    documents,
    key=lambda doc: cosine_similarity(query_embedding, doc["embedding"]),
    reverse=True,
)

for doc in ranked[:2]:
    print(doc["id"], doc["text"])
```

Real systems replace the fake vectors with an embedding model and replace the in-memory list with a vector store.

## Common Failure Modes

Poor chunking:

The answer exists, but the relevant text is split across chunks or buried inside a huge chunk.

Missing metadata:

The system retrieves the right meaning from the wrong source, tenant, date, or permission scope.

Wrong retrieval unit:

Embedding entire documents produces vague matches instead of precise evidence.

Too few results:

`top_k` is too small, so the needed context never reaches the LLM.

Too many results:

The LLM receives noisy context and answers from the wrong chunk.

No evaluation set:

The team changes embedding models, chunk sizes, or thresholds without measuring whether retrieval improved.

Only vector search:

Exact identifiers, product names, or error codes are missed because the system lacks keyword or hybrid search.

Stale index:

Documents changed, but old chunks are still in the vector database.

Permission leak:

The system retrieves text the user is not allowed to see.

## How To Evaluate Retrieval

Do not only test the final chatbot answer. Test retrieval directly.

Create a small evaluation table:

```text
Question | Expected source | Expected answer fact
```

Example:

```text
"How do I reset my password?"
Expected source: IT Handbook > Password Reset
Expected fact: Use the identity portal
```

Measure:

- Recall@k: did the expected chunk appear in the top k results?
- Precision: how many retrieved chunks were actually useful?
- MRR: how high was the first useful result ranked?
- answer groundedness: did the final answer use retrieved evidence?
- latency: how long did retrieval add?
- cost: how many embedding and generation tokens were used?

For agent developers, the most important question is:

```text
Did the agent retrieve the evidence needed before it answered or acted?
```

## Practical Defaults

Use these as starting points, then measure.

```text
Chunk size:          300-800 tokens
Chunk overlap:       50-150 tokens
Top-k before rerank: 10-30
Top-k after rerank:  3-8
Search type:         hybrid if exact terms matter
Metadata:            source, URL/path, section, timestamp, tenant, permission
Evaluation:          at least 20-50 representative questions
```

For a small learning project:

- use a hosted embedding model or a local sentence-transformer model,
- store vectors in memory, SQLite, Postgres with pgvector, Chroma, LanceDB, Qdrant, Weaviate, or Pinecone,
- build a tiny query script,
- inspect the retrieved chunks before connecting an LLM.

## Opinion

Embeddings and vector search are often presented as magic, but in real agent systems they are mostly an information-retrieval engineering problem.

The embedding model matters, but the bigger quality drivers are usually:

- clean source data,
- good chunk boundaries,
- useful metadata,
- correct permission filters,
- hybrid search for exact terms,
- reranking for precision,
- retrieval evaluation before launch.

For an AI agent developer, the right mindset is:

```text
The vector database is not the brain.
It is the agent's searchable memory and evidence index.
```

The LLM still decides how to use the retrieved context. Your job is to make sure the context is relevant, allowed, fresh, and traceable.

## Practice

Build a small semantic search script.

1. Choose 5-10 short documents.
2. Split them into chunks.
3. Generate embeddings for each chunk.
4. Store each chunk with metadata.
5. Embed a user query.
6. Retrieve the top 3 chunks.
7. Print the chunk text, score, and source.
8. Change the query wording and check whether the same useful chunks are found.
9. Add one exact identifier, such as `ERR-4017`, and test whether keyword or hybrid search is needed.

Write down:

- which chunk was retrieved first,
- whether the answer was actually present,
- what failed,
- what you changed,
- whether quality, cost, latency, or reliability improved.
